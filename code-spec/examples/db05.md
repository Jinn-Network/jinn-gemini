# Default Behavior Violation: Silent Catch Blocks

## ❌ Wrong

```typescript
// Violation 1: Completely silent
try {
  await riskyOperation();
} catch {
  // Silent failure - error disappears
}

// Violation 2: Empty catch
try {
  await riskyOperation();
} catch (e) {
  // Empty - same as silent
}

// Violation 3: Logged but not propagated (sometimes wrong)
try {
  await riskyOperation();
} catch (e) {
  workerLogger.error('Failed', { error: e });
  // Not re-thrown - caller thinks it succeeded!
}
```

## Why Silent Catches are Wrong

### The Problem

**Caller's perspective:**
```typescript
async function processData() {
  await riskyOperation(); // Actually failed silently
  await nextOperation();   // Proceeds with invalid state
  return "Success!";       // Returns success despite failure
}
```

**What the caller thinks:** "riskyOperation succeeded"

**Reality:** "riskyOperation threw an error, but it was silently caught"

### Real-World Impact

```typescript
// ❌ Silent catch
async function saveUserData(userId: string, data: any) {
  try {
    await database.save(userId, data);
  } catch {
    // Oops, save failed but we don't tell anyone
  }
  // Caller thinks save succeeded!
  await sendConfirmationEmail(userId); // Email sent despite failed save
  return { success: true }; // Lie!
}
```

**Result:** User gets confirmation email, but their data wasn't saved. **Data loss.**

### Debugging Nightmare

**Symptoms:**
- "It should work but doesn't"
- "Data is missing but no errors in logs"
- "Intermittent failures with no pattern"

**Root cause:** Silent catch swallowed the real error

## When is Catching Without Re-throwing OK?

### Case 1: Explicit Graceful Fallback

```typescript
// ✅ OK: Logged + explicit fallback
async function fetchWithFallback(url: string): Promise<Data | null> {
  try {
    return await fetch(url);
  } catch (error) {
    workerLogger.error('Fetch failed, returning null', {
      url,
      error: error instanceof Error ? error.message : String(error)
    });
    return null; // Explicit fallback, type reflects possibility
  }
}
```

**Why this is OK:**
1. Error is logged with context
2. Fallback strategy is explicit ("returning null")
3. Return type reflects failure possibility (`Data | null`)
4. Caller knows to handle `null` case

### Case 2: Non-Critical Operations

```typescript
// ✅ OK: Non-critical operation with logging
async function trackAnalytics(event: string) {
  try {
    await analyticsService.track(event);
  } catch (error) {
    workerLogger.warn('Analytics tracking failed (non-critical)', {
      event,
      error: error instanceof Error ? error.message : String(error)
    });
    // Don't re-throw - analytics failures shouldn't break main flow
  }
}
```

**Why this is OK:**
1. Operation is explicitly non-critical
2. Error is logged (as warning, not error)
3. Main operation can proceed without it
4. Documented that failures are acceptable

### Case 3: Retry Logic

```typescript
// ✅ OK: Catch to retry, re-throw if all retries fail
async function fetchWithRetry(url: string, maxRetries = 3): Promise<Data> {
  for (let i = 0; i < maxRetries; i++) {
    try {
      return await fetch(url);
    } catch (error) {
      if (i === maxRetries - 1) {
        // Last retry failed, re-throw
        workerLogger.error('Fetch failed after retries', {
          url,
          retries: maxRetries,
          error: error instanceof Error ? error.message : String(error)
        });
        throw error;
      }
      // Not last retry, log and continue
      workerLogger.warn('Fetch failed, retrying', {
        url,
        attempt: i + 1,
        maxRetries,
        error: error instanceof Error ? error.message : String(error)
      });
      await sleep(1000 * (i + 1)); // Exponential backoff
    }
  }
  throw new Error('Unreachable'); // TypeScript needs this
}
```

## Correct Versions

### Default: Re-throw

```typescript
// ✅ Log and re-throw (default pattern)
try {
  await riskyOperation();
} catch (error) {
  workerLogger.error('Risky operation failed', {
    error: error instanceof Error ? error.message : String(error)
  });
  throw error; // Let caller decide how to handle
}
```

### Graceful Handling

```typescript
// ✅ Log and handle explicitly
try {
  await riskyOperation();
} catch (error) {
  workerLogger.error('Risky operation failed, using fallback', {
    error: error instanceof Error ? error.message : String(error)
  });
  // Explicit recovery logic
  return fallbackValue;
}
```

## How to Fix Silent Catches

**Before:**
```typescript
try {
  await operation();
} catch {}
```

**After (Option 1: Re-throw):**
```typescript
try {
  await operation();
} catch (error) {
  workerLogger.error('Operation failed', { error });
  throw error;
}
```

**After (Option 2: Explicit Handling):**
```typescript
try {
  await operation();
} catch (error) {
  workerLogger.error('Operation failed, continuing with fallback', { error });
  return fallbackValue;
}
```

## Summary

**Never acceptable:**
- ❌ `catch { }` - No logging
- ❌ `catch (e) { }` - No logging
- ❌ Logging without deciding to re-throw or handle

**Always do:**
- ✅ Log with structured context
- ✅ Either re-throw OR explicitly handle with fallback
- ✅ Make it clear in logs what's happening

**Rule of thumb:** If you're not sure whether to re-throw, **re-throw**.
