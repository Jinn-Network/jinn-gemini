# Default Behavior Violation: Multiple Property Fallbacks

## ❌ Wrong

```typescript
// Violation: Checking 4 different property names
const input = signalCall.input || signalCall.arguments || signalCall.args || signalCall.result;
```

## What This Indicates

This pattern reveals that **there is no canonical data structure**. Different code paths are producing objects with different shapes:

```typescript
// Source A produces:
{ input: "data" }

// Source B produces:
{ arguments: "data" }

// Source C produces:
{ args: "data" }

// Source D produces:
{ result: "data" }
```

This violates the **orthodoxy objective**: there should be one obvious way to structure this data.

## Why This is a Problem

### 1. No Single Source of Truth

```typescript
// Developer 1 writes:
const data = call.input;

// Developer 2 writes:
const data = call.arguments;

// Developer 3 writes:
const data = call.args || call.input; // Trying to handle both

// Developer 4 writes:
const data = call.input || call.arguments || call.args || call.result; // Chaos
```

**Problem:** No one knows which property name is canonical. Code becomes increasingly defensive.

### 2. Maintenance Nightmare

When you need to change the data structure:

```typescript
// How do you migrate this?
const input = a || b || c || d;

// Which one is actually used in production?
// Do we need all four fallbacks?
// Can we remove any?
// Unknown!
```

### 3. Debugging Difficulty

```typescript
// Which property actually had the data?
console.log(input); // "data"
// But was it from .input, .arguments, .args, or .result?
```

### 4. Type Safety Lost

```typescript
// TypeScript can't help you
type ToolCall = {
  input?: any;
  arguments?: any;
  args?: any;
  result?: any;
};

// No way to enforce "exactly one of these"
```

## Correct Solution: Normalize at the Boundary

### Step 1: Create Canonical Structure

```typescript
// Define the canonical shape
type NormalizedToolCall = {
  input: any; // Only one property name
};
```

### Step 2: Normalize External Data

```typescript
// Normalize at the system boundary (once)
function normalizeToolCall(call: any): NormalizedToolCall {
  // Handle all legacy formats, but return canonical shape
  return {
    input: call.input || call.arguments || call.args || call.result
  };
}
```

### Step 3: Use Canonical Shape Internally

```typescript
// ✅ All internal code uses the canonical structure
const normalized = normalizeToolCall(signalCall);
const input = normalized.input; // Only one way to access

// Never need fallbacks internally
function processToolCall(call: NormalizedToolCall) {
  const input = call.input; // Always exists, always the same property
}
```

## Complete Example

### Before (Wrong)

```typescript
function extractSignalFromTelemetry(telemetry: any): FinalStatus | null {
  const toolCalls = telemetry?.toolCalls || telemetry?.tool_calls;
  if (!toolCalls) return null;

  const signalCall = toolCalls.find((call: any) =>
    call.name === 'finalize_job' || call.tool === 'finalize_job'
  );
  if (!signalCall) return null;

  // ❌ Multiple fallbacks scattered throughout code
  const input = signalCall.input || signalCall.arguments || signalCall.args || signalCall.result;
  if (!input) return null;

  const status = input.status;
  const message = input.message;

  return { status, message };
}
```

### After (Correct)

```typescript
// Define canonical structures
type NormalizedTelemetry = {
  toolCalls: NormalizedToolCall[];
};

type NormalizedToolCall = {
  name: string;
  input: any;
};

// Normalize at boundary
function normalizeTelemetry(telemetry: any): NormalizedTelemetry {
  const rawToolCalls = telemetry?.toolCalls || telemetry?.tool_calls || [];
  return {
    toolCalls: rawToolCalls.map((call: any) => ({
      name: call.name || call.tool || 'unknown',
      input: call.input || call.arguments || call.args || call.result || {}
    }))
  };
}

// ✅ Internal code uses canonical structure only
function extractSignalFromTelemetry(telemetry: any): FinalStatus | null {
  const normalized = normalizeTelemetry(telemetry);

  const signalCall = normalized.toolCalls.find(call =>
    call.name === 'finalize_job'
  );
  if (!signalCall) return null;

  // Only one way to access input
  const input = signalCall.input;
  if (!input) return null;

  const status = input.status;
  const message = input.message;

  return { status, message };
}
```

## Benefits of Normalization

✅ **Single source of truth** - One canonical structure

✅ **Easier to maintain** - Normalization logic in one place

✅ **Better debugging** - Know exactly what shape data has

✅ **Type safety** - TypeScript can enforce canonical structure

✅ **Simpler code** - No defensive fallbacks scattered everywhere

✅ **Clear migration path** - Update normalization function, not all call sites

## When You See Multiple Fallbacks

**It's a code smell that means:**

1. **No canonical structure exists** - Violates orthodoxy
2. **External data is inconsistent** - Need normalization layer
3. **Technical debt** - Multiple sources producing different shapes

**What to do:**

1. Identify all possible shapes
2. Define canonical shape
3. Create normalization function
4. Apply at system boundary
5. Update internal code to use canonical shape only
6. Document as default behavior
