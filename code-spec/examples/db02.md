# Default Behavior Violation: Using .catch() Instead of try/catch

## ❌ Wrong

```typescript
fetch(url).catch(err => console.log(err));
```

## Violations

1. ❌ **Uses `.catch()`** instead of `try/catch` block
2. ❌ **Uses `console.log`** instead of `workerLogger.error`
3. ❌ **No structured context** - Missing `url` in error context
4. ❌ **Error not propagated** - Silently swallows the error after logging

## Why Each Violation Matters

### 1. .catch() vs try/catch

**Problem:** Mixes promise chains with async/await idiom

```typescript
// ❌ Mixing paradigms
const data = await someAsyncOp();
fetch(url).catch(err => handle(err)); // Promise chain
const result = await anotherAsyncOp();
```

**Better:**
```typescript
// ✅ Consistent async/await
const data = await someAsyncOp();
try {
  await fetch(url);
} catch (err) {
  handle(err);
}
const result = await anotherAsyncOp();
```

### 2. console.log vs workerLogger

**Problem:** Unstructured, not observable in telemetry

```typescript
// ❌ Not queryable, not correlated with traces
console.log(err);

// ✅ Structured, queryable, observable
workerLogger.error('Fetch failed', { url, error: err });
```

### 3. Missing Context

**Problem:** Can't debug which URL failed

```typescript
// ❌ "An error occurred" - but which URL? Which operation?
console.log(err);

// ✅ Full context for debugging
workerLogger.error('Fetch failed', { url, operation: 'fetchData', error: err });
```

### 4. Silent Swallowing

**Problem:** Caller thinks operation succeeded

```typescript
// ❌ Error logged but swallowed
async function fetchData() {
  const result = await fetch(url).catch(err => {
    console.log(err);
    return null; // Swallowed, caller doesn't know it failed
  });
  return result; // Could be null due to error, but caller can't tell
}

// ✅ Error propagated
async function fetchData() {
  try {
    const result = await fetch(url);
    return result;
  } catch (error) {
    workerLogger.error('Fetch failed', { url, error });
    throw error; // Caller can decide how to handle
  }
}
```

## Correct Version

```typescript
async function fetchData(url: string): Promise<Response> {
  try {
    const response = await fetch(url);
    return response;
  } catch (error) {
    workerLogger.error('Fetch failed', {
      url,
      error: error instanceof Error ? error.message : String(error)
    });
    throw error;
  }
}
```

## Why This Matters

- `.catch()` creates inconsistency when rest of code uses async/await
- `console.log` bypasses structured logging and observability stack
- Missing context makes production debugging nearly impossible
- Swallowing errors masks failures and creates subtle bugs
