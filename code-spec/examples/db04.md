# Default Behavior Violation: Unstructured Logging

## ❌ Wrong

```typescript
// Violation 1: Passing error directly
catch (error) {
  workerLogger.error(error);
}

// Violation 2: String only, no context
catch (error) {
  workerLogger.error('Failed');
}

// Violation 3: String interpolation instead of object
catch (error) {
  workerLogger.error(`Failed: ${error}`);
}

// Violation 4: Missing structured context
catch (error) {
  workerLogger.error('Operation failed', error);
}
```

## What's Wrong

### Violation 1: Passing Error Directly

```typescript
// ❌ Error object passed as first argument
workerLogger.error(error);

// What gets logged:
// "[object Object]" or error.toString()
```

**Problem:** Logger expects string message + context object. Passing error directly:
- Loses error message formatting
- Loses context fields
- Not queryable by operation type

### Violation 2: String Only

```typescript
// ❌ No context about what failed
workerLogger.error('Failed');
```

**Problem:**
- Which operation failed?
- What were the inputs?
- What was the actual error?
- Can't query or filter logs

### Violation 3: String Interpolation

```typescript
// ❌ Error embedded in string
workerLogger.error(`Failed: ${error}`);
```

**Problem:**
- Not machine-parseable
- Can't query by error type
- Loses error.stack
- Loses structured fields

### Violation 4: Error as Second Argument

```typescript
// ❌ Error passed directly as second param
workerLogger.error('Operation failed', error);
```

**Problem:**
- Some loggers might accept this, but it's not structured
- Doesn't follow the pattern of context objects
- Loses ability to add other context fields

## Correct Versions

### Minimal Correct Logging

```typescript
catch (error) {
  workerLogger.error('Operation failed', {
    error: error instanceof Error ? error.message : String(error)
  });
}
```

### With Full Context

```typescript
catch (error) {
  workerLogger.error('Failed to fetch data', {
    url,
    jobId,
    error: error instanceof Error ? error.message : String(error),
    stack: error instanceof Error ? error.stack : undefined
  });
}
```

### With Operation Context

```typescript
catch (error) {
  workerLogger.error('Job processing failed', {
    jobId,
    step: 'execution',
    duration: Date.now() - startTime,
    error: error instanceof Error ? error.message : String(error),
    stack: error instanceof Error ? error.stack : undefined
  });
}
```

## Why instanceof Error Check Matters

```typescript
// ❌ Assumes error is Error object
catch (error) {
  workerLogger.error('Failed', {
    error: error.message  // Crashes if error is a string!
  });
}

// ✅ Handles any error type safely
catch (error) {
  workerLogger.error('Failed', {
    error: error instanceof Error ? error.message : String(error),
    stack: error instanceof Error ? error.stack : undefined
  });
}
```

**Why this matters:**
- Third-party code might throw strings: `throw "Something went wrong"`
- Promises might reject with non-Error values: `Promise.reject(404)`
- Network errors might be objects without Error prototype

## Good Context Fields

Include relevant debugging information:

```typescript
{
  // Operation identifiers
  jobId: string,
  requestId: string,
  userId: string,

  // Operation inputs
  url: string,
  params: object,

  // Operation state
  step: string,
  retryCount: number,
  duration: number,

  // Error details
  error: string,
  stack: string | undefined,
  errorCode: string | undefined
}
```

## Real Example: Good vs Bad

### Bad

```typescript
try {
  await dispatchParentJob(parentJobDefId);
} catch (e) {
  workerLogger.error(`Error dispatching parent job ${parentJobDefId}:`, e);
}
```

**Problems:**
- String interpolation
- Error passed directly
- Can't query by jobId

### Good

```typescript
try {
  await dispatchParentJob(parentJobDefId);
} catch (error) {
  workerLogger.error('Error dispatching parent job', {
    parentJobDefId,
    requestId,
    error: error instanceof Error ? error.message : String(error),
    stack: error instanceof Error ? error.stack : undefined
  });
}
```

**Benefits:**
- Structured context
- Queryable by parentJobDefId
- Safe error handling
- Includes stack trace
