# Default Behavior: Async Error Handling - Correct Implementation

## ✅ Correct

```typescript
async function deliverResult(requestId: string, data: any): Promise<void> {
  try {
    const tx = await deliverViaSafe(requestId, data);
    workerLogger.info('Result delivered successfully', {
      requestId,
      txHash: tx.hash
    });
  } catch (error) {
    workerLogger.error('Failed to deliver result', {
      requestId,
      error: error instanceof Error ? error.message : String(error),
      stack: error instanceof Error ? error.stack : undefined
    });
    throw error;
  }
}
```

## Why This Follows the Default Behavior

✅ **Uses try/catch block** - Explicit error boundary, not .catch() promise chain

✅ **Uses workerLogger.error()** - Structured logging, not console.*

✅ **Structured context object** - Includes `requestId` for debugging and correlation

✅ **Error serialization** - Handles both Error objects and other types safely:
```typescript
error: error instanceof Error ? error.message : String(error)
```

✅ **Re-throws error** - Allows caller to decide if error is recoverable

## Key Pattern Elements

1. **Try/catch** - Wraps async operation explicitly
2. **Logger** - `workerLogger` for observability (OpenTelemetry compatible)
3. **Context** - Relevant debugging info as structured object
4. **Propagation** - Error re-thrown to caller (or explicitly handled)

## Alternative: Graceful Fallback

```typescript
async function fetchJobWithRetry(jobId: string): Promise<Job | null> {
  try {
    const response = await fetch(`/api/jobs/${jobId}`);
    if (!response.ok) {
      throw new Error(`HTTP ${response.status}: ${response.statusText}`);
    }
    return await response.json();
  } catch (error) {
    workerLogger.error('Failed to fetch job, returning null', {
      jobId,
      error: error instanceof Error ? error.message : String(error)
    });
    return null; // Explicit fallback instead of re-throw
  }
}
```

**Note:** Not re-throwing is acceptable when:
1. The error is handled gracefully (explicit fallback like `return null`)
2. The fallback strategy is logged clearly ("returning null")
3. The return type reflects the possibility of failure (`Job | null`)

## Common Usage Pattern

```typescript
async function processJob(jobId: string): Promise<void> {
  try {
    const job = await fetchJob(jobId);
    const result = await executeJob(job);
    await saveResult(jobId, result);

    workerLogger.info('Job processed successfully', { jobId });
  } catch (error) {
    workerLogger.error('Job processing failed', {
      jobId,
      error: error instanceof Error ? error.message : String(error),
      stack: error instanceof Error ? error.stack : undefined
    });
    throw error;
  }
}
```
