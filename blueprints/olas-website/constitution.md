# Olas Website Constitution

These are the core, immutable principles that guide the Olas website venture. All decisions, requirements, and implementations must align with these principles.

## Principles

### Accuracy and Authority

**Assertion:**
The website is the authoritative source of truth for information about the Olas protocol, ecosystem, and community.

**Examples:**

| Do | Don't |
|---|---|
| Verify all technical content against official documentation and source code | Publish unverified or speculative information about protocol features |
| Link directly to canonical sources (GitHub, docs) for technical details | Rely on third-party interpretations without verification |
| Clearly distinguish between current features and roadmap items | Present future plans as current capabilities |

**Commentary:**

As the official website for Olas, accuracy is paramount. Users, developers, and stakeholders rely on this website for authoritative information. All content must be verifiable, up-to-date, and aligned with the actual state of the protocol and ecosystem. This principle ensures trust and credibility.

---

### Community-Centric Design

**Assertion:**
The website serves the diverse needs of the Olas community, prioritizing accessibility, clarity, and inclusion over aesthetic preferences.

**Examples:**

| Do | Don't |
|---|---|
| Design navigation for multiple user personas (developers, operators, users) | Optimize layout for a single audience type |
| Ensure WCAG 2.1 AA accessibility compliance | Use design patterns that exclude users with disabilities |
| Provide content in multiple formats (text, video, interactive demos) | Limit content to a single medium |

**Commentary:**

The Olas community includes developers building services, operators running agents, researchers studying the protocol, and users interacting with autonomous services. The website must accommodate all these personas with clear, accessible information paths. Accessibility is not optional - it is a fundamental requirement.

---

### Open and Transparent

**Assertion:**
The website's development, content updates, and decision-making processes are public and participatory.

**Examples:**

| Do | Don't |
|---|---|
| Accept community contributions via GitHub pull requests | Restrict content changes to a small, closed team |
| Document major design decisions in public issues | Make architectural changes without community visibility |
| Maintain a public changelog of website updates | Deploy changes without announcing them |

**Commentary:**

In alignment with Olas's decentralized ethos, the website itself should be developed transparently. The repository at https://github.com/oaksprout/olas-website-1 serves as the public workspace where anyone can propose improvements, report issues, or contribute content. This openness fosters trust and encourages community participation.

---

### Performance and Reliability

**Assertion:**
The website loads quickly, functions reliably, and remains available across diverse network conditions and devices.

**Examples:**

| Do | Don't |
|---|---|
| Target sub-2s initial page load on 3G connections | Require high-bandwidth connections for basic navigation |
| Implement progressive enhancement with core content accessible without JavaScript | Build a single-page app that breaks without JavaScript |
| Monitor uptime and performance continuously | Deploy without performance testing or monitoring |

**Commentary:**

The website represents the Olas protocol to a global audience with varying internet infrastructure. Fast, reliable access is essential for adoption and engagement. Performance is not just a technical metric - it directly impacts user experience and the protocol's reach.
