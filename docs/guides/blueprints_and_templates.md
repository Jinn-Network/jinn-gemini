# Jinn Blueprint & Template Guide

This document defines the core concepts of the Jinn Agent System: **Templates**, **Venture Blueprints**, and the **Runtime Blueprint**.

## 1. Core Definitions

### 1.1 The Template ("The Product Container")
A **Template** is the distinct, sellable unit in the Jinn Marketplace. It is a **container** that holds the venture's logic along with the necessary configuration for the marketplace and the user.

*   **Role:** Defines the product interface (inputs, outputs, price) and wraps the core logic.
*   **Location:** `blueprints/*.json` (Source of truth).
*   **Components:**
    *   `inputSchema`: JSON Schema for user configuration.
    *   `outputSpec`: Definition of deliverable structures.
    *   `price`: Cost in Wei/USDC.
    *   **Venture Blueprint:** The `invariants` array within the template (see below).

### 1.2 The Venture Blueprint ("The Logic / Root")
The **Venture Blueprint** is the set of **Goal Invariants** specific to a venture. It is **not** a hybrid; it is the pure business logic and requirements for that specific undertaking.

*   **Role:** Defines *what* success looks like for this specific venture.
*   **Location:** The `invariants` array inside a Template, or a standalone `blueprint.json` generated by a factory.
*   **Content:** Only contains **Goal Invariants** (`GOAL-*`). It does **not** contain System, Safety, or Protocol invariants.

> **Key Concept:** The Venture Blueprint is the "Root" logic for the venture. The System wraps around it.

### 1.3 The Runtime Blueprint ("The Instruction Set")
The **Runtime Blueprint** is the completely assembled object that the Jinn Worker constructs in memory and sends to the Agent.

*   **Role:** The actual prompt/instructions the agent follows.
*   **Composition:** `BlueprintBuilder` assembles this by layering:
    1.  **System Framework:** `SystemInvariantProvider` (Identity, Safety)
    2.  **Venture Blueprint:** `GoalInvariantProvider` (The specific logic from 1.2)
    3.  **Context:** `JobContextProvider` (Live state, files, git status)
    4.  **Protocol Rules:** `CoordinationInvariantProvider`, `StrategyInvariantProvider`

## 2. Writing a Template

A Template is a JSON file that packages a Venture Blueprint.

**File:** `blueprints/example-venture.json`

```json
{
  // --- Marketplace Configuration ---
  "name": "Code Health Auditor",
  "inputSchema": { ... },
  "priceWei": "...",

  // --- The Venture Blueprint (Logic) ---
  "invariants": [
    {
      "id": "GOAL-001",
      "domain": "goal",
      "invariant": "Run a security audit on {{repoUrl}}.",
      "measurement": "Verify audit-report.md exists."
    },
    {
      "id": "GOAL-002",
      "domain": "goal",
      "invariant": "Fix critical issues automatically.",
      "measurement": "Verify no CRITICAL issues remain."
    }
  ]
}
```

### Best Practices
*   **Pure Goals:** Only define `GOAL` invariants. Do not try to redefine `SYSTEM` rules (e.g., "You are an AI"). The System will handle that.
*   **Variable Substitution:** Use `{{inputVar}}` handlebars for values defined in `inputSchema`.
*   **Measurability:** Every invariant must have a clear `measurement` string.
*   **Tool Policy:** Templates declare tool policy via a single `tools` list with `required` annotations. Universal tools are injected by the platform and must not be listed in templates.

### Tool Policy Fields

Templates may include tool policy fields at the top level (preferred) or inside `templateMeta`:

```json
{
  "tools": [
    { "name": "blog_create_post", "required": true },
    { "name": "process_branch", "required": true },
    { "name": "write_file" },
    { "name": "replace" }
  ]
}
```

**Rules:**
- Tools marked `required: true` are always enabled for the workstream (in addition to universal tools).
- All listed tools are part of the whitelist for child jobs. Child jobs may only request tools from this list.
- Do **not** include universal tools (e.g., `dispatch_new_job`, `create_artifact`, `google_web_search`). The platform injects them automatically.
 

### Input Schema Design

**Use flat structures** for inputSchema. Avoid nested objects.

✅ **Good (flat):**
```json
{
  "inputSchema": {
    "properties": {
      "blogName": { "type": "string" },
      "targetAudience": { "type": "string" },
      "brandColor": { "type": "string", "default": "colors appropriate for the brand" }
    },
    "required": ["blogName", "targetAudience"]
  }
}
```

❌ **Avoid (nested):**
```json
{
  "inputSchema": {
    "properties": {
      "blogSpec": {
        "properties": { "name": {}, "colors": { "primary": {} } }
      }
    }
  }
}
```

**Why flat?**
- Simpler variable substitution: `{{blogName}}` vs `{{blogSpec.name}}`
- Easier default handling in the gateway
- Better readability for both humans and agents

**Defaults:** Each optional field should have a natural language `default` that makes sense if not provided:
```json
"brandVoice": { "type": "string", "default": "a tone appropriate for the audience" }
```

## 3. Generating Venture Blueprints (The Factory Pattern)

When a "Foundry" agent creates a new venture, it generates a **Venture Blueprint**.

### The "Minimalism Principle"
The generated file (`blueprint.json`) should **only** contain the Venture Blueprint (the Goal Invariants).

*   **DO:** Generate a JSON with an `invariants` array of `GOAL` items.
*   **DO NOT:** Copy System Invariants ("SYS-001") into the file.
*   **DO NOT:** Copy Protocol Invariants ("COORD-001").

The Jinn Worker will automatically verify and wrap this generated Venture Blueprint with the necessary System and Protocol layers at runtime.

**Example Generated File (`venture-package/blueprint.json`):**
```json
{
  "invariants": [
    {
      "id": "GOAL-001",
      "domain": "goal",
      "invariant": "Scrape https://target.com and analyze trends.",
      "measurement": "Verify analysis.md contains trend data."
    }
  ]
}
```

## 4. Summary Table

| Term | Analogy | Content |
| :--- | :--- | :--- |
| **Venture Blueprint** | The Application Code | Goal Invariants (`invariants` array) |
| **Template** | The Docker Image | Venture Blueprint + Inputs + Price + Metadata |
| **System** | The OS / Kernel | System Invariants + Safety + Coordination |
| **Runtime Blueprint** | The Running Process | System + Venture Blueprint + Context |
