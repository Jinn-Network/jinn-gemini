# Rule 3: Never Silently Discard Errors in Financial or Blockchain Contexts

## Context

In systems handling real cryptocurrency and making irreversible blockchain transactions, **silent failures are catastrophic**. When errors are swallowed without logging or propagation, critical issues become invisible:

- **Lost funds**: Failed transfers may go unnoticed until funds are missing
- **Debugging impossible**: No logs = no way to trace what went wrong
- **Cascading failures**: Silent errors in one operation cause downstream failures
- **Incident response blocked**: Can't fix what you can't see

**This rule applies to:**
- Token transfers (OLAS, ETH)
- Safe transactions (deliveries, mech operations)
- RPC calls to blockchain nodes
- IPFS uploads for on-chain deliveries
- Database writes for on-chain job tracking

**What "silent" means:**
- Empty `catch` blocks with no logging
- `.catch(() => null)` fallbacks without logging
- Returning default values on error without logging
- Swallowing errors in Promise chains

**Exceptions (allowed silent handling):**
- Non-critical background operations (e.g., telemetry upload failures)
- Explicit graceful degradation with logging at lower severity
- Worker loop resilience (log and continue, don't crash entire system)

## Test Cases

### Case 1: Token Transfer Failures

#### ❌ Violates Rule

```typescript
// File: worker/funding.ts

// ❌ VIOLATION: Empty catch block
export async function transferOlas(
  to: string,
  amount: bigint
): Promise<void> {
  try {
    const olasToken = new ethers.Contract(OLAS_ADDRESS, OlasABI, signer);
    const tx = await olasToken.transfer(to, amount);
    await tx.wait();
  } catch (error) {
    // Silent failure: funds may not have transferred, no indication
  }
}
```

```typescript
// ❌ VIOLATION: Fallback without logging
export async function transferOlas(
  to: string,
  amount: bigint
): Promise<boolean> {
  const olasToken = new ethers.Contract(OLAS_ADDRESS, OlasABI, signer);
  const tx = await olasToken.transfer(to, amount).catch(() => null);

  return tx !== null; // Silent failure: why did it fail? unknown
}
```

```typescript
// ❌ VIOLATION: Swallowed in Promise chain
export function transferOlas(to: string, amount: bigint) {
  const olasToken = new ethers.Contract(OLAS_ADDRESS, OlasABI, signer);

  return olasToken
    .transfer(to, amount)
    .then(tx => tx.wait())
    .then(() => true)
    .catch(() => false); // Silent failure: error lost
}
```

**Why this is dangerous:**
- User thinks transfer succeeded (function returns normally)
- Actual transfer failed (insufficient balance, network error, etc.)
- No logs = impossible to debug why funds didn't arrive
- Incident may not be discovered until funds are needed

#### ✅ Follows Rule

```typescript
// File: worker/funding.ts

// ✅ CORRECT: Explicit error handling with logging
export async function transferOlas(
  to: string,
  amount: bigint
): Promise<void> {
  try {
    const olasToken = new ethers.Contract(OLAS_ADDRESS, OlasABI, signer);

    logger.info('Transferring OLAS', {
      to,
      amount: ethers.formatUnits(amount, 18)
    });

    const tx = await olasToken.transfer(to, amount);
    const receipt = await tx.wait();

    logger.info('OLAS transfer confirmed', {
      to,
      amount: ethers.formatUnits(amount, 18),
      txHash: receipt.hash
    });
  } catch (error) {
    // ✅ Log error with full context
    logger.error('OLAS transfer failed', {
      to,
      amount: ethers.formatUnits(amount, 18),
      error: error instanceof Error ? error.message : String(error)
    });

    // ✅ Re-throw to propagate to caller
    throw error;
  }
}
```

### Case 2: Mech Delivery Failures

#### ❌ Violates Rule

```typescript
// File: worker/mech_worker.ts

// ❌ VIOLATION: Silent catch in critical path
export async function deliverResult(
  requestId: string,
  resultContent: any
) {
  try {
    await deliverViaSafe({
      requestId,
      resultContent,
      targetMechAddress: MECH_ADDRESS,
      safeAddress: SAFE_ADDRESS,
      wait: true
    });
  } catch (error) {
    // Silent failure: delivery failed but no indication
    // Result: Job stays in "IN_PROGRESS" forever
  }
}
```

**Why this is dangerous:**
- Job claim remains in `IN_PROGRESS` state indefinitely
- No delivery transaction submitted (request stays undelivered)
- No logs to understand why delivery failed
- User doesn't receive result, no way to retry

#### ✅ Follows Rule

```typescript
// File: worker/mech_worker.ts

// ✅ CORRECT: Explicit error handling with recovery
export async function deliverResult(
  requestId: string,
  resultContent: any,
  metadata?: any
) {
  try {
    const deliveryResult = await deliverViaSafe({
      requestId,
      resultContent,
      targetMechAddress: MECH_ADDRESS,
      safeAddress: SAFE_ADDRESS,
      wait: true
    });

    workerLogger.info('Delivery succeeded', {
      jobName: metadata?.jobName,
      requestId,
      txHash: deliveryResult.tx_hash,
      status: deliveryResult.status
    });

    return deliveryResult;
  } catch (error) {
    // ✅ Log error with full context
    workerLogger.error('Safe delivery failed', {
      jobName: metadata?.jobName,
      requestId,
      error: serializeError(error)
    });

    // ✅ Update job status to FAILED (prevent stuck IN_PROGRESS)
    try {
      await apiCreateJobReport(requestId, {
        status: 'FAILED',
        error_message: error instanceof Error ? error.message : String(error),
        error_type: 'DELIVERY_ERROR',
        final_output: typeof resultContent === 'string'
          ? resultContent
          : JSON.stringify(resultContent)
      });
    } catch (reportErr) {
      // ✅ Even this failure is logged
      workerLogger.error('Failed to record FAILED status', {
        requestId,
        error: serializeError(reportErr)
      });
    }

    // ✅ Re-throw to surface to caller
    throw error;
  }
}
```

### Case 3: RPC Call Failures

#### ❌ Violates Rule

```typescript
// File: worker/chain_state.ts

// ❌ VIOLATION: Silent RPC failures
export async function getUndeliveredRequests(
  mechAddress: string
): Promise<string[]> {
  try {
    const contract = new ethers.Contract(mechAddress, MechABI, provider);
    const ids = await contract.getUndeliveredRequestIds(100, 0);
    return ids;
  } catch {
    return []; // Silent failure: was there a network error? unknown
  }
}
```

**Why this is dangerous:**
- RPC node may be down, rate-limiting, or returning errors
- Caller thinks there are no undelivered requests (empty array)
- System appears healthy but isn't processing work
- No visibility into RPC reliability issues

#### ✅ Follows Rule

```typescript
// File: worker/chain_state.ts

// ✅ CORRECT: Log RPC errors, use typed error handling
export async function getUndeliveredRequests(
  mechAddress: string
): Promise<string[]> {
  try {
    const contract = new ethers.Contract(mechAddress, MechABI, provider);
    const ids = await contract.getUndeliveredRequestIds(100, 0);

    logger.debug('Fetched undelivered requests', {
      mechAddress,
      count: ids.length
    });

    return ids;
  } catch (error) {
    // ✅ Log error with context
    logger.error('Failed to fetch undelivered requests', {
      mechAddress,
      error: error instanceof Error ? error.message : String(error),
      // Include error type for debugging
      errorType: error instanceof Error ? error.constructor.name : typeof error
    });

    // ✅ Re-throw to force caller to handle
    throw error;
  }
}
```

### Case 4: IPFS Upload Failures

#### ❌ Violates Rule

```typescript
// File: gemini-agent/mcp/tools/create_artifact.ts

// ❌ VIOLATION: Silent IPFS upload failure
export async function uploadToIPFS(content: string): Promise<string | null> {
  const cid = await fetch('https://registry.autonolas.tech/api/v0/add', {
    method: 'POST',
    body: JSON.stringify({ content })
  })
    .then(res => res.json())
    .then(data => data.cid)
    .catch(() => null); // Silent failure: why did upload fail?

  return cid;
}
```

**Why this is dangerous:**
- Upload may fail due to network issues, API limits, invalid content
- Caller receives `null` with no context why
- On-chain delivery may require IPFS hash (delivery fails)
- No visibility into IPFS infrastructure reliability

#### ✅ Follows Rule

```typescript
// File: gemini-agent/mcp/tools/create_artifact.ts

// ✅ CORRECT: Explicit IPFS error handling
export async function uploadToIPFS(content: string): Promise<string> {
  try {
    logger.debug('Uploading to IPFS', {
      contentLength: content.length
    });

    const response = await fetch(
      'https://registry.autonolas.tech/api/v0/add',
      {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ content })
      }
    );

    if (!response.ok) {
      throw new Error(
        `IPFS upload failed: HTTP ${response.status} ${response.statusText}`
      );
    }

    const data = await response.json();

    if (!data.cid) {
      throw new Error('IPFS response missing CID field');
    }

    logger.info('IPFS upload successful', {
      cid: data.cid,
      contentLength: content.length
    });

    return data.cid;
  } catch (error) {
    // ✅ Log error with context
    logger.error('IPFS upload failed', {
      contentLength: content.length,
      error: error instanceof Error ? error.message : String(error)
    });

    // ✅ Re-throw to propagate to caller
    throw error;
  }
}
```

## Key Principles

### 1. **Log Before Swallowing**
- If you must handle an error without re-throwing, log it first
- Include full context: operation, parameters, error message
- Use appropriate log level (`error` for failures, `warn` for degradation)

### 2. **Prefer Throwing to Fallbacks**
- Default: Re-throw errors to propagate up the stack
- Only use fallbacks when there's a legitimate degraded mode
- Document why the fallback is safe

### 3. **Structured Error Information**
- Use structured logging (`logger.error('message', { context })`)
- Don't log raw error objects (may contain secrets in `.config`)
- Serialize errors safely: `error instanceof Error ? error.message : String(error)`

### 4. **Fail Fast in Critical Paths**
- Financial operations should throw on error (don't continue)
- Let the caller decide retry logic vs. abort
- Worker loop: catch at top level, log, continue (don't let one job crash worker)

### 5. **Context-Specific Error Handling**

**Financial operations** (Rule applies):
```typescript
// ✅ MUST log and throw
try {
  await transfer(to, amount);
} catch (error) {
  logger.error('Transfer failed', { to, amount, error });
  throw error; // ✅ Propagate
}
```

**Non-critical background tasks** (Exception allowed):
```typescript
// ✅ Acceptable: Telemetry upload can degrade gracefully
try {
  await uploadTelemetry(data);
} catch (error) {
  logger.warn('Telemetry upload failed', { error }); // ✅ Logged
  // Continue execution (don't throw)
}
```

## Enforcement

### Code Review Checklist

Before merging code with error handling:
- [ ] No empty `catch` blocks in financial/blockchain code
- [ ] No `.catch(() => null)` without logging in critical paths
- [ ] Errors logged with structured context
- [ ] Errors re-thrown or handled with explicit recovery logic
- [ ] Error messages include operation, parameters, and error details

### Linter Rules (Future)

```javascript
// eslint rule to detect empty catches
{
  "no-empty": ["error", { "allowEmptyCatch": false }],
  "no-unused-vars": ["error", { "caughtErrors": "all" }]
}
```

## Application to AI Code Generation

When AI generates error handling code:
1. **Default to explicit logging**: Never generate empty catches
2. **Include context in logs**: AI should capture operation parameters
3. **Prefer throwing to silent fallbacks**: Let caller decide recovery
4. **Learn from successful patterns**: Reuse `serializeError()`, structured logging

The "Never silently discard" rule ensures AI-generated error handling remains observable and debuggable.
